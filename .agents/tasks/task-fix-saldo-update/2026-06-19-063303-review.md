# Dirty-read protection and saldo awal parity for balance reports

The change addresses stale balance reports after transaction edits. The root cause was twofold: `KDB.getAll()` could return stale Firebase data before an async write completes, and `renderPrintBundle()` was computing saldo without the opening balance (`saldo_awal`) that `getFinancialData()` already included. A secondary fix converts `l.debit`/`l.kredit` through `parseFloat()` in four accumulation sites to prevent string concatenation when values are stored as strings.

**Watch for:** The dirty-read window is time-based (10 seconds) — if Firebase write takes longer (e.g., slow connection), stale data can still win (likely). The `get()` single-document method has the same race condition but was not patched (confirmed). Saldo awal logic is copy-pasted across two functions with no shared extraction (confirmed).

## High-level view

The Firebase data layer now tracks "dirty" items in localStorage and in-memory, then merges local versions into `getAll()` results when they are less than 10 seconds old. This prevents a fast `getAll()` from overwriting freshly-saved data that Firebase hasn't acknowledged yet. The approach is simple and avoids introducing a queue or write-through cache, but it trades correctness guarantees for a time-based heuristic.

`renderPrintBundle()` now loads opening balances (`saldo_awal_YYYY`) before summing journal lines, matching the logic `getFinancialData()` already had. This closes the discrepancy where the print report showed different totals than the dashboard. The code is duplicated verbatim rather than extracted into a shared function, creating a maintenance risk for future drift.

The `parseFloat()` fix is applied across all four journal-line accumulation points (`getFinancialData`, `renderPrintBundle` balance section, `renderPrintBundle` cash flow section, `buildDashboardExtras`). This guards against debit/kredit values stored as strings, which would produce concatenation instead of addition (e.g., `"5000" + 3000 = "50003000"`).

## Dirty-read protection via time-based merge

The race condition: `save()` writes to localStorage, then awaits `setDoc()` to Firebase. If another code path calls `getAll()` before Firebase reflects the write, the Firestore snapshot overwrites the fresh local data. The fix introduces a two-layer dirty tracking:

```js
// In save():
_klset('k_' + col + '_dirty_' + id, Date.now());
this._recentSaves[col][id] = Date.now();

// In getAll():
if (dirtyTime && (now - dirtyTime) < 10000) {
  // Use local version
}
```

The localStorage flag survives page reloads; the in-memory `_recentSaves` map drives the lookup loop. However, `getAll()` iterates only over `Object.keys(this._recentSaves[col])` — so after a page reload, `_recentSaves` is an empty object, and no dirty items are discovered even though their localStorage flags still exist. This means a save-then-immediate-refresh scenario (common on mobile) leaves the dirty items unprotected. The fix would be to scan localStorage for dirty keys matching the collection on `getAll()` when `_recentSaves[col]` is empty.

The `get(col, id)` method (line 98) reads a single document from Firebase and returns it directly without checking the dirty flag. If a caller does `save(col, id, data)` then immediately `get(col, id)` (which happens in some edit flows), the returned value could be stale Firebase data. The fix should mirror the dirty check: if `k_{col}_dirty_{id}` exists and is within window, return the local version instead.

## Saldo awal parity in renderPrintBundle

`renderPrintBundle()` produces the print-ready financial statements (Laba Rugi, Neraca, Arus Kas). Before this fix, it computed account balances solely from journal lines, omitting the opening balance that `getFinancialData()` (used by the interactive dashboard) included. The result: the print report showed different balance totals than the dashboard after saldo awal was configured.

The added block is a line-for-line copy of `getFinancialData()`'s saldo awal handling: fetch setting, iterate keys, classify debit-normal vs credit-normal using `tipe`, `kategori` substring, or leading account code digit, then add to the appropriate side. The classification heuristic (`kode.charAt(0) === '1'` for assets, `'5'`/`'6'` for expenses) is an implicit contract with the chart of accounts — if accounts are renumbered without updating this logic, balances silently misclassify.

## parseFloat guards against string concatenation

Journal line values (`l.debit`, `l.kredit`) can be strings when stored to/from localStorage or Firebase (JSON parse of user input). The original code used `l.debit || 0` which evaluates to the string `"5000"` (truthy) instead of coercing to a number. The `+=` operator then concatenates: `0 + "5000"` → `"05000"`, `"05000" + 3000` → `"050003000"`. Wrapping in `parseFloat()` eliminates this class of bug across all four accumulation sites.
