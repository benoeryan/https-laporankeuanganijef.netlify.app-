# PWA shell, PDF export, and FCM push notifications for Keuangan IJEF

This change makes the financial reporting app installable as a PWA on Android and iOS, adds a PDF download button for the full financial report bundle (using html2canvas + jsPDF from CDN), and integrates Firebase Cloud Messaging for push notifications in both foreground and background. The approach follows an existing sister app (hr-legal-app) as a template for the service worker architecture and FCM wiring.

**Watch for:** Race condition between `initNotifikasi` and `initFCM` means FCM silently skips initialization when notification settings haven't loaded yet (confirmed). The `gcm_sender_id` in manifest.json doesn't match the project's actual `messagingSenderId`, which will break push delivery on some browsers (confirmed). Two separate `loadScript` functions are defined — one scoped inside `downloadPdfLaporan`, one at module level — creating a shadowing situation that works by accident today but is fragile (confirmed).

## High-level view

The PDF feature dynamically loads html2canvas and jsPDF on first click, renders the `.print-bundle-section` DOM element to a canvas, slices it into A4-page-sized chunks, and saves the result. Text and table rows that land on slice boundaries will be cut mid-line — acceptable for v1 but visible on dense financial tables.

The PWA shell consists of a manifest, a caching service worker (`sw.js`), and a separate FCM service worker (`firebase-messaging-sw.js`). The caching SW uses a network-first strategy but excludes JS and HTML from the cache, meaning the app itself won't load offline — the offline story is limited to images and fonts.

FCM initialization loads the Firebase compat SDK alongside the app's existing ES module Firebase, creating two independent Firebase app instances. The critical timing issue is that `initFCM()` is called immediately after `initNotifikasi()` starts but before its promise resolves, so `_notifEnabled` is still `false` when FCM checks it — push notifications will never activate.

<details>
<summary>Issues (6)</summary>

1. **FCM skipped due to race condition** — `initFCM()` is called before `initNotifikasi()` resolves, so `_notifEnabled` is always `false` at FCM init time. Move `initFCM()` inside the `.then()` callback of `initNotifikasi()`.
2. **Wrong `gcm_sender_id` in manifest** — manifest.json uses `"103953800507"` (copied from hr-legal-app) but this project's `messagingSenderId` is `"48180557823"`. Replace with the correct sender ID or the browser may reject push subscription.
3. **Activate handler nukes all caches unconditionally** — `sw.js` deletes every cache key on activate. Every SW update causes a full cache wipe with no warm-up; users on flaky connections lose all cached assets immediately.
4. **No offline capability for the app shell** — JS and HTML are excluded from caching. The app cannot launch offline. If PWA installability suggests offline use, this is a gap users will hit.
5. **Dual `loadScript` definitions** — A function-scoped `loadScript` inside `downloadPdfLaporan` and a module-level `loadScript` at the bottom of the file. The module-level one is used by `initFCM`. If code ordering changes or another caller uses the name, the shadowing will produce confusing bugs.
6. **FCM tokens never cleaned up** — Tokens accumulate in `k_fcm_tokens` as users switch devices or tokens rotate. The collection grows unbounded. Add a timestamp check and prune stale entries on login.

## FCM initialization race

`buildApp()` calls `initNotifikasi().then(...)` and immediately calls `initFCM()` on the next line without awaiting the notification initialization:

```javascript
initNotifikasi().then(function(){ updateNotifBadge(); });
initFCM();
```

`initNotifikasi` is async — it reads `notif_enabled` from Firestore via `KDB.getSetting` and assigns the result to `_notifEnabled`. But `initFCM()` runs before the Firestore read completes. The first guard in `initFCM` is `if (!_notifEnabled) return;`, which will always exit early because `_notifEnabled` still holds its initial value of `false`.

The fix is to call `initFCM()` inside the resolved `.then()` of `initNotifikasi()`, or `await initNotifikasi()` before calling `initFCM()`.

## Manifest `gcm_sender_id` mismatch

The manifest contains:

```json
"gcm_sender_id": "103953800507"
```

This value was copied from the hr-legal-app reference, but this project's Firebase config has `messagingSenderId: "48180557823"`. The `gcm_sender_id` field is used by Chrome to associate the push subscription with the correct Firebase project. A mismatch means the browser may subscribe to the wrong sender, causing push messages from this project's FCM to never arrive.

## PDF pagination boundary cuts

The pagination approach renders the full report to one large canvas, then slices vertically into A4-proportioned strips. Each strip becomes a JPEG added to a new PDF page. If text or table rows land on a slice boundary, they'll be cut mid-line. For a financial report with dense tables, this will happen on most multi-page exports. Content-aware page breaking would require DOM-level splitting rather than canvas slicing.

## Dual Firebase SDK loading

The existing `keuangan-firebase.js` loads Firebase v10.7.1 via ES module `import()` and initializes a named app instance (`'keuangan'`). The new `initFCM()` loads Firebase v10.12.2 compat SDK via script tags and calls `firebase.initializeApp(KFB_CONFIG)` (default app name). This results in two Firebase app instances in memory — doubled initialization overhead, and a version skew (10.7.1 vs 10.12.2) that could surface subtle behavioral differences if the two ever interact through shared Firestore state.

## Token storage without cleanup

FCM tokens are stored in `k_fcm_tokens` with a document ID of `username_hash(token)`. Tokens are never pruned. FCM rotates tokens periodically, and users switching devices leave orphaned entries. Over time this inflates the collection and causes wasted FCM send attempts (which return errors but don't fail visibly).
