# Bidirectional forecast integration with transaction pages

The change adds navigation buttons between the Forecast Pembayaran/Penerimaan monitoring pages and the Permohonan Dana/Dana Masuk transaction pages, making the flow bidirectional. Overdue forecast items get action buttons that navigate to the transaction page and prefill the form with the forecast data. The Permohonan Dana and Dana Masuk pages gain a forecast button with a live overdue badge count. The approach is straightforward DOM manipulation with inline `onclick` handlers and a `setTimeout`-based prefill after navigation.

**Watch for:** XSS via incomplete escaping in inline onclick handlers (confirmed), race condition in the 300ms setTimeout prefill (likely), and redundant database reads that duplicate work already done elsewhere in the render cycle (confirmed).

## High-level view

The forecast table gains an "Aksi" column whose buttons pass forecast data into inline `onclick` handlers. These handlers call `prefillPermohonanFromForecast` or `prefillDanaMasukFromForecast`, which navigate to the transaction page and fill form fields after a fixed 300ms delay. The escaping applied to the data before injection into the onclick string is incomplete — single quotes and double quotes are handled, but backslashes, newlines, and other control characters are not, creating an XSS surface when forecast names come from user-controlled database records.

The overdue badge counts on Permohonan Dana and Dana Masuk are computed by fetching the full `utangpiutang` and `permohonan`/`danamasuk` collections again, even though these same collections are already fetched earlier in the same render function. This doubles the IndexedDB reads on every page render for no functional benefit.

The prefill functions rely on a hardcoded 300ms `setTimeout` to wait for the target page to render before setting form values. If the render takes longer (large dataset, slow device), the prefill silently fails — form fields remain empty with no indication to the user that something went wrong.

<details>
<summary>Issues (4)</summary>

1. **XSS in inline onclick handlers** — `escapedNama` and `escapedRef` only escape single quotes and `&quot;` but not backslashes or control characters. A forecast record with a backslash-single-quote sequence in its name bypasses the escaping. Sanitize all special characters or pass data via `data-*` attributes with event delegation instead of inline handlers.
2. **Brittle 300ms setTimeout prefill** — The prefill functions assume the target page renders within 300ms. On slow devices or large datasets this races. Use a callback, event, or poll for the target element's existence instead of a fixed delay.
3. **Duplicate database reads for overdue count** — `renderPermohonanDana` already fetches `permohonan` into `list` and could derive overdue counts from it rather than issuing a separate `KDB.getAll('permohonan')` call. Same pattern in `renderDanaMasuk` with `danamasuk`.
4. **Prefill doesn't trigger input events** — Setting `.value` directly won't fire `oninput`/`onchange` handlers. The thousand-separator formatting (`formatNominalInput`) and any reactive validation attached to those inputs won't activate, potentially causing the raw numeric value to be submitted without formatting or validation.

## XSS surface in inline onclick string construction

The action buttons in `renderForecastWithChart` build onclick handlers by interpolating escaped values into a string template:

```javascript
var escapedNama = (x.nama||'').replace(/'/g, "\\'").replace(/"/g, '&quot;');
```

This escapes `'` → `\'` and `"` → `&quot;`, but the data originates from user-controlled database records (utangpiutang entries). A record with `nama` set to `\\'` would produce `\\\\'` after the replace, which in the onclick context becomes a literal backslash followed by an unescaped quote — breaking out of the string. More practically, newline characters (`\n`) in the nama field would break the JavaScript parser inside the attribute.

The safer pattern used elsewhere in this codebase is to store data in `data-*` attributes and attach a single delegated event listener, or to use `JSON.stringify` for the full serialization. The current approach is a latent injection vector.

## Race condition in setTimeout-based prefill

Both `prefillPermohonanFromForecast` and `prefillDanaMasukFromForecast` call `navigate(...)` then `setTimeout(..., 300)`. The `navigate` function calls `renderSection` which is `async` — it awaits database reads and builds the DOM. If those reads take longer than 300ms (plausible on mobile with large IndexedDB stores), `document.getElementById('pd-pemohon')` returns `null`, the `if (elPemohon)` guard skips the assignment, and the form opens empty with no error feedback to the user.

A more robust pattern: poll for the element's existence with a short interval and a timeout ceiling, or hook into the render completion.

## Duplicate data fetching for overdue badge

In `renderPermohonanDana`, `list` is already the full `permohonan` collection filtered by sumber. The new code fetches `KDB.getAll('permohonan')` a second time to compute `pdPendingForecast`. Similarly `KDB.getAll('utangpiutang')` is fetched again despite being available (or trivially reusable) from the forecast rendering path. For `renderDanaMasuk` the same duplication occurs with `danamasuk` and `utangpiutang`.

This isn't a correctness bug but doubles IndexedDB I/O on pages that are already doing multiple async reads. On collections with hundreds of records this adds perceptible latency.

## Prefill bypasses input formatting hooks

The existing codebase attaches `oninput="formatNominalInput(this)"` to monetary inputs. The prefill functions set `.value` directly without dispatching an `input` event. The displayed value will be correct (because `formatNominalValue` is used), but any side effects attached to those input handlers — validation state, derived calculations, dirty-form detection — won't fire. If a future change adds validation that depends on the `input` event having been triggered, these prefilled forms will silently bypass it.
