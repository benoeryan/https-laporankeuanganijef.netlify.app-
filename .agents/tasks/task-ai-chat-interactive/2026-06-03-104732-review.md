# Multi-turn conversational AI chat with action editing

The AI chat evolves from a single-shot request/response into a multi-turn conversation. The Netlify Function now accepts conversation history and uses Gemini's `system_instruction` field to separate persona instructions from user content. The frontend gains an edit-before-confirm workflow for AI-proposed actions, three new action types (`koreksi_jurnal`, `analisis_akun`, `laporan_ringkasan`), completeness warnings, and randomized follow-up messages after every action outcome.

**Watch for:** XSS in the edit/confirmation flow where user-edited values are injected into `innerHTML` without escaping (confirmed). The `koreksi_jurnal` handler saves a corrected journal without verifying debit/kredit balance, unlike the `jurnal` handler which validates before write (confirmed). History content from prior assistant messages (which may contain `###ACTION:...###` markers) is sent back to Gemini unfiltered, which could confuse the model into emitting duplicate actions (likely).

## High-level view

The Netlify Function moves the system prompt into Gemini's `system_instruction` field and builds a multi-turn `contents` array from the last 10 history messages. The body size limit doubles to 100KB. This increases the attack surface for prompt injection via replayed assistant messages that contain action markers.

The frontend adds an inline edit form that lets users modify action fields before executing. After saving edits, the modified action is re-rendered via `renderActionConfirmation` and injected into the DOM via `innerHTML`. Because values from `<input>` elements are interpolated into raw HTML without escaping, this is an XSS vector.

Three new action types extend the executor. `koreksi_jurnal` does an in-place update of an existing journal entry but skips the debit/kredit balance check that guards the `jurnal` creation path. `analisis_akun` and `laporan_ringkasan` are read-only client-side data renders.

4. **No content-length sanitization on history items** â€” The backend caps the history array to 10 items but does not limit per-item string length. 5. **Duplicate confirmation card on edit-reconfirm** â€” After `simpanEditAIAction`, a new confirmation card is appended but the original remains visible in the DOM with stale values. Remove or disable the previous confirmation on edit.

## XSS in the action confirmation and edit flow

`renderActionConfirmation` builds HTML by string concatenation using values like `action.keterangan`, `action.id`, `action.alasan`, and account codes. These values originate either from Gemini's JSON output (which is user-influenceable via prompt injection) or from the edit form's `<input>` elements:

```javascript
html += '<b>Koreksi Jurnal</b><br>'
  + 'ID Jurnal: ' + (action.id||'-') + '<br>'
  + 'Keterangan Baru: ' + (action.keterangan||'-') + '<br>'
```

A `keterangan` containing `<img src=x onerror=alert(1)>` would execute. The codebase already has an `escapeHtml` function used for chat bubble display text, but it's not applied anywhere in `renderActionConfirmation` or `editAIAction`. Since the confirmation card is rendered after Gemini responds (or after the user edits and saves), a crafted Gemini response becomes a stored XSS triggered by viewing the confirmation.

## koreksi_jurnal saves without balance validation

The `jurnal` creation path validates:

```javascript
var totalD = 0, totalK = 0;
(action.lines||[]).forEach(function(l) { totalD += l.debit||0; totalK += l.kredit||0; });
if (Math.abs(totalD - totalK) > 1) {
  showAlert('Jurnal tidak balance!...', 'danger');
  return;
}
```

The `koreksi_jurnal` path computes `tdKoreksi` and `tkKoreksi` but only uses them to set `totalDebit`/`totalKredit` fields on the saved object â€” it never checks whether they're equal before calling `KDB.save`. A correction with mismatched debits and credits will be persisted and subsequently flagged by the anomaly checker, creating confusion about when the imbalance was introduced.

## History replay and prompt confusion

When an assistant message containing `###ACTION:{"type":"jurnal",...}###` is pushed into `_aiChatHistory`, that verbatim string (including the action marker) is later sent back to Gemini as a prior `model` turn. The system prompt tells Gemini to emit `###ACTION:...###` when proposing actions; seeing its own prior action markers in history may lead it to re-emit them or interpret them as already-executed commands. Stripping action markers from the `content` field before pushing to `_aiChatHistory` would prevent this while preserving the displayable explanation text.

## Edit flow DOM management

After `simpanEditAIAction` removes the `#ai-edit-action` form, it appends a fresh confirmation card at the bottom of the chat. The original confirmation card (with the old values) remains in the DOM above. Both have `onclick="eksekusiAIAction()"` pointing to the same global `_pendingAIAction` reference, so clicking either executes the latest version â€” but two confirmation cards with different displayed values is confusing. The old card should be removed or visually disabled when the edit is saved.

## Body size limit and history growth

The backend raised `MAX_BODY_SIZE` from 50KB to 100KB. With 10 history messages and a 10KB context string, worst-case is around 65KB for typical messages. However, `analisis_akun` and `laporan_ringkasan` follow-up messages pushed to `_aiChatHistory` can be multi-line reports. If several such reports appear in the last 10 messages, the payload could approach the limit. A per-message truncation (e.g., 2000 chars) on history items sent to the backend would add a safety margin.
